# Demo 04：RAG Agent 本地知识库问答

## 学习目标

理解 RAG（检索增强生成）的核心思想：先检索，再回答。

## RAG 是什么？

```
RAG = Retrieval（检索）+ Augmented（增强）+ Generation（生成）

用户问："什么是 Agent？"
          │
          ▼
   ① Retrieval（检索）
      从本地文档中找到相关段落
          │
          ▼
   ② Augmented（增强）
      把检索到的段落和用户问题拼接在一起
          │
          ▼
   ③ Generation（生成）
      模型基于拼接后的内容生成回答（还能引用来源）
```

## 和普通聊天的区别

| | 普通聊天 | RAG |
|---|---|---|
| 知识来源 | 模型训练数据（可能过时） | 本地文档（最新、可控） |
| 答案准确性 | 可能编造（幻觉） | 基于文档，更可靠 |
| 可否溯源 | 不知道答案从哪来 | 能引用具体文档 |
| 更新方式 | 需要重新训练模型 | 更新文档即可 |

## 本 Demo 的简化设计

**不使用向量数据库**，而是用最简单的**关键词匹配**：

1. 文档按 `##` 标题拆分成 chunk（段落块）
2. 用户提问拆成关键词
3. 统计每个 chunk 中关键词出现次数
4. 返回匹配最多的 Top-3 chunk

这虽然不是最优方案，但能让你直观理解检索的本质。后续可升级为 embedding + 向量搜索。

## Chunk 是什么？

长文档没法整篇塞进模型上下文（token 限制）。Chunk 就是把文档切成小块：

```
原始文档（1500 字）
  → Chunk 1：什么是 Agent（300 字）
  → Chunk 2：Agent 三要素（400 字）
  → Chunk 3：ReAct 模式（350 字）
  → Chunk 4：与 LLM 区别（450 字）
```

检索时只取出最相关的几个 chunk，而不是整篇文档。

## RAG 的知识泄露风险

如果检索到的 chunk 中包含敏感信息（如 API Key、内部数据），模型可能会在回复中暴露这些信息。所以：
- 知识库文档要经过审查
- 敏感信息不要放入可被检索的文档
- 生产环境需要权限控制
